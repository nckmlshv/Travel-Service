# Travel-Service

Сервис позволяет пользователю спланировать поездку: подобрать билеты, отели, трансферы, активности. Все бронирования синхронизируются в едином интерфейсе, система выдаёт рекомендации на основе интересов и истории.
 
## Функциональные требования

### Управление поездкой и совместной работой

**FR-1.** Система должна позволять создавать поездку и добавлять в неё нескольких участников с разными ролями.  
**FR-2.** Система должна обеспечивать совместное редактирование плана поездки несколькими пользователями.

### Работа с элементами поездки

**FR-3.** Система должна позволять добавлять, редактировать и удалять элементы поездки:
- рейсы,
- отели,
- активности,
- трансферы.

**FR-4.** Система должна хранить текущее (выбранное) состояние каждого элемента поездки.

### Альтернативные варианты

**FR-5.** Система должна позволять предлагать альтернативные варианты элементов (без перезаписи текущего).

**FR-6.** Система должна хранить и отображать:
- выбранный вариант,
- все предложенные альтернативы.

### Синхронизация и оффлайн

**FR-7.** Система должна синхронизировать изменения между участниками, включая изменения, сделанные оффлайн.

### Конфликты и merge

**FR-8.** Система должна выявлять конфликты при конкурентных изменениях.

**FR-9.** Система должна поддерживать merge-стратегии:
- автоматическое объединение независимых изменений,
- сохранение нескольких вариантов,
- ручное разрешение конфликта.

### Логика маршрута и интеграции

**FR-10.** Система должна:
- проверять согласованность маршрута (время, пересадки, пересечения),
- учитывать изменения из внешних API (задержки, отмены).

## Нефункциональные требования

### Консистентность и данные

**NFR-1.** Система должна обеспечивать согласованное состояние поездки для всех участников после синхронизации.

**NFR-2.** Система не должна терять пользовательские изменения при конкурентной работе или оффлайн-режиме.

### Производительность

**NFR-3.** Изменения должны становиться доступными другим участникам с минимальной задержкой.

### Надёжность

**NFR-4.** Система должна продолжать работу при частичной недоступности внешних API.

**NFR-5.** Система должна гарантировать сохранность данных при сбоях сети или клиента.

### Безопасность

**NFR-6.** Система должна защищать персональные и платёжные данные пользователей.

**NFR-7.** Система должна разграничивать доступ к данным между участниками поездки.

### Масштабируемость

**NFR-8.** Система должна масштабироваться по количеству пользователей, поездок и операций синхронизации.

## Особенности проектирования

- Система должна поддерживать совместную работу пользователей с единым состоянием поездки, ролями и одновременным редактированием без потери данных.

- Система должна работать с внешними сервисами (рейсы, отели), учитывая различия форматов, ограничения API и возможные ошибки.

- Система должна обеспечивать безопасную обработку и хранение платёжных и паспортных данных с разграничением доступа.

- Система должна корректно обрабатывать время, тайм-зоны и международные форматы данных (даты, валюты, адреса).

- Система должна выявлять конфликты расписаний, пересечения и логические ошибки маршрута и уведомлять пользователей.

- Система должна предоставлять персонализированные рекомендации без раскрытия персональных данных между участниками.

- Система должна быть устойчивой к сбоям внешних API и обеспечивать сохранность уже полученных и введённых данных.
