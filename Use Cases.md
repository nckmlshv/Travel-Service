# Use Cases

| UC-01 | Синхронизация данных и обнаружение конфликтов |
|---|---|
| **Акторы** | Пользователь, Сервис синхронизации |
| **Предусловия** | 1. Пользователь авторизован.<br>2. На клиенте есть локальные события в оффлайн-буфере.<br>|
| **Триггер** | 1. Восстановление сетевого соединения. |
| **Основной поток** | 1. Пользователь передает пакет изменений вместе с историей версий в сервис синхронизации.<br>2. Сервис сравнивает историю правок пользователя с текущим состоянием данных на сервере.<br>3. Сервис выявляет несовместимые правки.<br>4. Сервис проверяет, не устарела ли локальная версия слишком сильно (например, объект изменения уже удален).<br>5. Сервис сохраняет входящие правки в защищенный лог, не изменяя основной план.<br>6. Регистрируется запись в таблице конфликтов.<br>7. Сервис отправляет уведомление всем участникам группы о необходимости ручного вмешательства. |
| **Постусловия** | 1. Создана запись об инциденте.<br>2. Основная ветка плана осталась без изменений.<br>3. Участники группы уведомлены о наличии конфликта в планах поездки.<br>4. Локальные изменения пользователя не пропали, а ожидают ручного слияния. |
| **Исключения** | 1. Сервер временно не может обработать пакет.<br>1.1. Запрос ставится в очередь на повторные попытки на стороне клиента с увеличивающейся задержкой.<br><br>4. Данные клиента основаны на слишком старой версии, которая уже была сильно изменена.<br>4.1. Система отклоняет локальный пакет событий и загружает пользователю актуальную версию.<br>4.2. Система информирует пользователя о том, что его локальные изменения отклонены, и объясняет причины. |
| **Связанные требования** | FR-7. Система должна синхронизировать изменения между участниками, включая изменения, сделанные оффлайн.<br>FR-8. Система должна выявлять конфликты при конкурентных изменениях.<br>FR-9. Система должна поддерживать merge-стратегии: автоматическое объединение независимых изменений, сохранение нескольких вариантов, ручное разрешение конфликта.<br>NFR-1. Система должна обеспечивать согласованное состояние поездки для всех участников после синхронизации.<br>NFR-2. Система не должна терять пользовательские изменения при конкурентной работе или оффлайн-режиме.<br>NFR-5. Система должна гарантировать сохранность данных при сбоях сети или клиента. |

| UC-02 | Ручное разрешение конфликта версий |
|---|---|
| **Акторы** | Организатор поездки / Участник с правами редактора, Сервис синхронизации |
| **Предусловия** | 1. Наличие записи со статусом конфликта в планах поездки.<br>2. Пользователь обладает правами записи в текущей поездке. |
| **Основной поток** | 1. Пользователь открывает список активных противоречий в поездке.<br>2. Система отображает сравнительный вид: текущая версия плана против предложенных изменений.<br>3. Пользователь выбирает одну из версий как приоритетную.<br>4. Система создает запись о слиянии, которая объединяет разрозненные ветки истории.<br>5. Система закрывает инцидент противоречия.<br>6. Система обновляет основной план для всех участников группы.<br>7. Система рассылает обновленное состояние всем активным пользователям. |
| **Постусловия** | 1. Противоречие устранено у всех участников группы способом, который выбрал пользователь.<br>2. Основная ветка плана обновлена. |
| **Исключения** | 3. Потеря связи в момент выбора. (шаг 3)<br>3.1. Запрос ставится в очередь на повторные попытки на стороне клиента с увеличивающейся задержкой.<br><br>4. Конфликт уже разрешен другим участником.<br>4.1. Система блокирует попытку повторного слияния.<br>4.2. Система уведомляет пользователя о том, что слияние уже выполнено другим пользователем. |
| **Связанные требования** | FR-8. Система должна выявлять конфликты при конкурентных изменениях.<br>FR-9. Система должна поддерживать merge-стратегии: автоматическое объединение независимых изменений, сохранение нескольких вариантов, ручное разрешение конфликта.<br>NFR-1. Система должна обеспечивать согласованное состояние поездки для всех участников после синхронизации. |

| UC-03 | Надежное бронирование через внешние API |
|---|---|
| **Акторы** | 1. Пользователь с правами бронирования.<br>2. Сервис бронирования.<br>3. Защищенное хранилище личных данных.<br>4. Внешний сервис услуг (билеты, отели и т.д.). |
| **Предусловия** | 1. Элемент маршрута утвержден.<br>2. В защищенном хранилище имеются актуальные паспортные данные пользователя. |
| **Основной поток** | 1. Пользователь инициирует процесс бронирования конкретной позиции.<br>2. Система блокирует возможность редактирования этой позиции другими участниками группы.<br>3. Система запрашивает у защищенного хранилища временный доступ к зашифрованным данным пользователя.<br>4. Система передает паспортные данные и параметры бронирования внешнему сервису через защищенный канал и ожидает подтверждения.<br>5. Система сохраняет номер бронирования и электронные билеты в плане.<br>6. Система снимает блокировку и меняет статус позиции на подтвержденный для всей группы. |
| **Постусловия** | 1. Подтверждение брони получено.<br>2. Данные в плане зафиксированы и защищены от правок. |
| **Исключения** | 4. Внешнее API не отвечает или вернуло ошибку.<br>4.1. Сессия бронирования сбрасывается, защищенный канал для передачи личных данных блокируется.<br>4.2. Блокировка с позиции в плане снимается, статус откатывается к состоянию «Доступно для бронирования».<br>4.3. Пользователь получает уведомление о том, что бронирование не удалось, так как внешний сервис временно недоступен, средства не списаны.<br><br>4. Ошибка валидации паспортных данных у внешнего сервиса.<br>4.1. Сессия бронирования сбрасывается, защищенный канал для передачи личных данных блокируется.<br>4.2. Блокировка с позиции в плане снимается, статус откатывается к состоянию «Доступно для бронирования».<br>4.3. Пользователь получает уведомление о том, что бронирование не удалось, так как в паспортных данных есть ошибка, средства не списаны.<br><br>4. Цена услуги у внешнего сервиса изменилась в процессе бронирования. (шаг 4)<br>4.1. Сессия бронирования сбрасывается, защищенный канал для передачи личных данных блокируется.<br>4.2. Блокировка с позиции в плане снимается, статус откатывается к состоянию «Доступно для бронирования».<br>4.3. Пользователь получает уведомление о том, что бронирование не удалось, так как стоимость услуги изменилась, средства не списаны. |
| **Связанные требования** | FR-4. Система должна хранить текущее (выбранное) состояние каждого элемента поездки.<br>FR-10. Система должна: проверять согласованность маршрута (время, пересадки, пересечения), учитывать изменения из внешних API (задержки, отмены).<br>NFR-4. Система должна продолжать работу при частичной недоступности внешних API.<br>NFR-6. Система должна защищать персональные и платёжные данные пользователей.<br>NFR-7. Система должна разграничивать доступ к данным между участниками поездки. |
